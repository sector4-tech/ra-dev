//===== rAthena Documentation ================================
//= Voice Chat System
//===== Description: =========================================
//= Proximity voice chat — server, client DLL, item license,
//= GM moderation commands.
//============================================================


===========================
| 1. Overview              |
===========================

The voice chat system is composed of three cooperating parts:

  Map server  (rAthena)        — emits player/position events via UDP
       ↓
  Voice server (standalone)    — relays Opus audio between connected DLL clients
       ↑
  Voice client DLL (in-game)   — captures mic, plays back received audio


  Map  ──UDP voice_bridge──▶  Voice server  ◀──WebSocket──▶  Client DLL


===========================
| 2. Files                 |
===========================

- src/voice/server.cpp ......... Voice server (audio relay, ban/license logic).
- src/map/voice_bridge.{cpp,hpp} UDP sender used by map server to talk to the
                                 voice server (join/leave/pos/admin commands).
- voice-chat-client/src/* ...... Client DLL (capture, playback, overlay).
- conf/voice_athena.conf ....... Voice server runtime config.
- sql-files/voice_chat.sql ..... voice_bans, voice_licenses, voice_blocks
                                 tables (also auto-created at startup).


===========================
| 3. GM Atcommands         |
===========================

@voicemute <player> [+time]
@voiceunmute <player>

  In-memory mute by char_id. The player can connect and hear others, but
  their audio is dropped server-side. Mutes are NOT persisted — they clear
  on voice server restart.

  Examples:
    @voicemute Albert            -> permanent mute
    @voicemute Albert +30n       -> 30 minute mute
    @voiceunmute Albert

@voiceban <player> [+time]
@voiceunban <player>

  Persistent ban keyed on account_id (a player cannot bypass by switching
  characters). Stored in the voice_bans table. Player stays connected and
  can hear, but audio TX is blocked silently. Supports offline players via
  char-name lookup.

  Examples:
    @voiceban Albert             -> permanent ban
    @voiceban Albert +7d         -> 7 day ban
    @voiceban OfflinePlayer +1d  -> works even if player is offline
    @voiceunban Albert

@voicegrant <player> [+time]
@voicerevoke <player>

  Grant or revoke a voice license. Only meaningful when the
  voice_license_required config is enabled (see Section 5). Stored in
  voice_licenses table — keyed on account_id, one license covers all
  characters on that account.

  Examples:
    @voicegrant Albert           -> permanent license (free for life)
    @voicegrant Albert +30d      -> 30 day license
    @voicerevoke Albert          -> remove the license

@reloadvoiceconf
@reloadvoicedb

  Reload conf/voice_athena.conf or the voice-related DB rules without
  restarting the voice server. License-required toggle takes effect on
  every active session immediately (no relog required).

Time format
-----------
The [+time] argument uses rAthena's solve_time() syntax:
    y/a = year   m = month   d/j = day   h = hour   n/mn = minute   s = second
    +1h    1 hour
    +30n   30 minutes
    +7d    7 days
    +1m    1 month


===========================
| 4. Script Commands       |
===========================

voicegrant(<duration_sec>);
voicerevoke {<account_id>};

  Used inside item or NPC scripts. Sends a UDP command to the voice
  server. The duration is in seconds; 0 means permanent.

  Both push 1 on success (UDP packet queued) or 0 if the voice bridge
  is not ready (voice server unreachable). Scripts can branch on the
  result to handle voice-server outages gracefully.

  Item script (simplest — accepts that item is lost on voice outage):

    - Id: 31000
      AegisName: Voice_Mobile_Perm
      Name: Voice Mobile (Permanent)
      Type: Usable
      Buy: 1000000
      Script: |
        voicegrant(0);

  NPC script (charge-on-success — recommended for production):

      prontera,150,150,4   script   Voice Shop   100,{
          mes "Buy a permanent voice license for 1,000,000z?";
          if (select("Yes:No") == 2) close;
          if (Zeny < 1000000) { mes "Not enough zeny."; close; }
          if (voicegrant(0) == 0) {
              mes "Voice server is unavailable. Please try again later.";
              close;
          }
          Zeny -= 1000000;
          mes "Permanent voice license activated.";
          close;
      }

  voicerevoke without an argument acts on the attached player; with an
  account_id it can be used from a non-player script context.


===========================
| 5. License Mode          |
===========================

The license feature is OFF by default. To enable it, edit
conf/voice_athena.conf:

    // 0 = anyone can use voice
    // 1 = require an active voice_licenses entry
    voice_license_required: 1

Then in-game:

    @reloadvoiceconf

When the flag flips from 0 to 1, the voice server walks every active
session and notifies the DLL:
  - players with an active license     -> license_granted    (mic stays on)
  - players without an active license  -> license_required   (mic shown off)

When the flag flips from 1 to 0, every active session receives
license_granted (mic re-enabled) — no relog required.

Persistence
-----------
voice_licenses rows are stored with an optional expires_at column.
NULL = permanent. The voice server's maintenance loop expires timed
licenses every 10 seconds and pushes a license_required event to the
affected player so the overlay updates immediately.

===========================
| 5b. Per-player Block     |
===========================

Players can block individual voices on their own — no GM action needed.
This is the voice equivalent of an "ignore list":

  - Asymmetric: the blocker stops hearing the blocked player. The blocked
    player can still hear the blocker (and is not notified).
  - Keyed on account_id so it follows the blocked player across character
    switches: blocking "Albert" automatically silences "AlbertAlt" on the
    same account.
  - Persistent in the voice_blocks DB table — survives relog and reinstall
    of the client DLL. Cross-device sync is automatic.
  - Server-side filter in should_forward(): audio for a blocked sender is
    never even forwarded to the blocker's DLL, saving bandwidth and making
    the block impossible to bypass with a modified client.

Client UX
---------
The overlay's player list provides a Mute / Unmute toggle which calls the
DLL's mute_player(char_id) / unmute_player(char_id). Internally the DLL
sends a block_add / block_remove WS message to the voice server and
optimistically updates its local UI list. The server stores the block
keyed on account_id and pushes the authoritative list back on every
auth_ok via a your_blocks message.

Limits
------
The block list only flags blocked players whose char_id is online at auth
time. If the blocked account comes online later with a different character,
the server still filters their audio correctly, but the overlay's
"(Blocked)" indicator won't update until the blocker relogs.

Block options (conf/voice_athena.conf)
--------------------------------------
  voice_block_bidirectional   1 = A blocks B → neither hears the other.
                              0 = asymmetric (only the blocker is affected).
  voice_block_alert_threshold N = alert online GMs when a player is blocked
                              by N distinct accounts (toxic detector). 0 = off.
  voice_ignore_sync           1 = a player's text /ex ignore also voice-blocks
                              that player (and /in removes it). 0 = independent.

GM block queries
----------------
  @voiceblockers <player>   How many players have voice-blocked <player>.
                            Works offline (matches by stored name).
  @voiceblocks <player>     List who <player> has blocked. Online only.

Toxic alert
-----------
When voice_block_alert_threshold is set and a player crosses it, the voice
server pushes a one-time block_alert to the map server, which broadcasts a
line to every online GM that holds the receive-requests permission. The
alert fires once per account per server uptime.

===========================
| 6. Bans                  |
===========================

voice_bans rows are stored with an optional banned_until column.
NULL = permanent. Same maintenance loop expires timed bans and pushes
admin_unbanned to the affected player so the mic icon clears
without needing a relog.

A banned player can still connect and HEAR everyone — only their
outgoing audio is silenced (the mic icon in their overlay shows OFF).
This is intentional: it avoids "why am I being disconnected?" support
tickets while still preventing voice-grief.


===========================
| 7. Flood Protection      |
===========================

The voice server tracks per-session audio rate. If a client sends
~30 consecutive rate-limited frames within 1 second, the server bans
the ACCOUNT (not the IP) for 5 minutes and tells the DLL via a
flood_banned message containing the remaining duration. The DLL's
reconnect loop then waits out the ban before trying again, so a
flood-banned client does not spam reconnects.

IP is intentionally NOT used as the ban key — families on the same
NAT would otherwise be collaterally punished.


===========================
| 8. SQL Setup             |
===========================

Run this once when first deploying the voice server (or just let the
voice server auto-create the tables on first start):

    sql-files/voice_chat.sq

It contains all three tables (voice_bans, voice_licenses, voice_blocks).
All are auto-created via CREATE TABLE IF NOT EXISTS at voice server
startup, so importing the file is optional.
